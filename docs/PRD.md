# PRD: 반도체 시료 생산주문관리 시스템 (SampleOrderSystem)

## 1. 개요

### 1.1 배경
가상의 반도체 회사 **S-Semi**는 다양한 종류의 반도체 시료(Sample)를 생산하여 연구소, 팹리스(Fabless) 업체,
대학 연구실 등의 고객에게 납품한다. 시료는 주문이 들어오면 웨이퍼 공정 설비를 통해 제작되고, 검수를 거쳐
고객에게 출고된다.

최근 주문량이 급증하면서 다음과 같은 문제가 발생했다.
- 주문 처리 여부를 파악하기 어려움
- 공정 예약 후 완성 시점을 알 수 없음
- 재고가 충분한데도 불필요한 추가 공정이 진행됨

엑셀과 메모장 기반 관리로 인해 실수가 잦고 재고·공정 현황을 한눈에 파악할 수 없어, 이를 해결하기 위한
**"반도체 시료 생산주문관리 시스템"** 을 개발한다.

### 1.2 목적
- 시료 등록/조회, 주문 접수/승인/거절, 생산, 출고에 이르는 전체 흐름을 하나의 시스템에서 일관되게 관리
- 재고와 생산 현황을 실시간으로 모니터링하여 의사결정을 지원
- 담당자의 수기 처리로 인한 실수를 제거하고 처리 이력을 시스템으로 추적

### 1.3 시스템 형태
- **콘솔(Console) 기반** 애플리케이션
- 담당자가 메뉴 선택 및 명령을 직접 입력하여 조작
- 데이터 영속성 필요: 애플리케이션을 재시작해도 기존 데이터(시료, 주문, 재고, 생산 큐 상태 등)가 유지되어야 함

## 2. 역할 및 흐름

### 2.1 역할
| 역할 | 설명 |
|---|---|
| 고객 | 필요한 시료를 이메일 등으로 요청하는 시료 요청자 |
| 주문 담당자 | 고객 요청에 맞게 주문서를 작성하고 주문서 관리를 담당 |
| 생산 담당자 | 개발 시료 등록, 주문 수신 후 승인/거절, 시료 생산을 담당 |

### 2.2 역할별 흐름
```
고객 --(시료 요청)--> 주문 담당자 --(주문서 전달)--> 생산 담당자
                ^                                        |
                └───────────────(승인 / 거절)────────────┘
```
1. 고객이 필요한 시료를 이메일 등으로 요청
2. 주문 담당자가 요청에 맞게 주문서를 작성(시스템에 주문 등록)
3. 생산 담당자가 주문을 수신하여 승인 또는 거절 처리

### 2.3 전체 프로세스 흐름 (상태 전이 다이어그램)
```
[주문 접수]
  RESERVED
     |
     v
 <승인 여부>
   |         \
 거절        승인
   |            \
   v              v
REJECTED      <재고 확인>
                 |          \
              재고 충분     재고 부족
                 |               \
                 v                 v
           CONFIRMED          PRODUCING (생산 요청)
          (출고 준비)               |
                 |             생산 완료
                 v                 |
           RELEASED  <-----------+
          (출고 처리)
                 |
                 v
             [고객 도달]
```

## 3. 주문 상태 정의

모든 주문은 아래 상태 중 하나를 가진다.

| 상태 | 의미 |
|---|---|
| RESERVED | 주문 접수 |
| REJECTED | 주문 거절 (정상 흐름 외의 상태이며, 모니터링 대상에서 제외) |
| PRODUCING | 주문 승인 완료 및 재고 부족으로 생산 중 |
| CONFIRMED | 주문 승인 완료 및 출고 대기 중 |
| RELEASED | 출고 완료 |

- 정상 흐름: `RESERVED → (CONFIRMED | PRODUCING) → CONFIRMED → RELEASED`
- 예외 흐름: `RESERVED → REJECTED` (종단 상태, 이후 전이 없음)
- `PRODUCING` 상태의 주문은 생산이 완료되는 즉시 `CONFIRMED`로 전환된다.

## 4. 기능 명세

### 4.1 메인 메뉴
- 기능별 선택 화면을 표시하고, 전체 시료에 대한 요약 정보(등록 시료 수, 총 재고, 전체 주문 수, 생산라인 대기 건수 등)를 함께 노출한다.

| 메뉴 | 설명 |
|---|---|
| 시료 관리 | 신규 시료 등록, 목록 조회, 이름 검색 |
| 주문 (접수/승인/거절) | 고객 주문 접수 및 생산 라인 담당자의 승인·거절 처리 |
| 모니터링 | 상태별 주문 수 및 시료별 재고 현황 확인 |
| 출고 처리 | CONFIRMED 상태 주문에 대한 출고 실행 |
| 생산 라인 | 현재 생산 중인 시료 및 대기 중인 생산 큐 확인 |

### 4.2 시료 관리
시료(Sample)는 시스템의 가장 기본이 되는 단위이며, 각 시료는 고유한 이름과 속성을 가진다.
**시스템에 등록된 시료만 주문이 가능하다.**

#### 4.2.1 시료 속성
| 속성 | 설명 |
|---|---|
| 시료 ID | 고유 식별자 |
| 이름 | 시료명 |
| 평균 생산시간 | 시료 1개(개당)를 생산하는 데 소요되는 평균 시간 |
| 수율 | 정상적인 시료 수 / 총 생산 시료 수 (예: 100개 중 정상 90개 → 0.9) |

#### 4.2.2 하위 기능
| 기능 | 설명 |
|---|---|
| 시료 등록 | 신규 시료를 시스템에 추가 (시료 ID, 이름, 평균 생산시간, 수율 입력) |
| 시료 조회 | 등록된 모든 시료 목록과 현재 재고 수량을 함께 표시 |
| 시료 검색 | 이름 등 속성으로 특정 시료를 검색 |

### 4.3 시료 주문 (접수)
고객이 시료를 요청하면 주문 담당자가 주문을 생성한다.

#### 4.3.1 시료 예약(주문 접수)
- 고객이 원하는 시료와 수량을 주문한다.
- 예약 시점의 주문 상태는 `RESERVED`이다.

**예약 시 입력 값**
- 시료 ID
- 고객명
- 주문 수량

### 4.4 주문 승인/거절
접수된 `RESERVED` 상태의 주문 목록을 확인하고, 특정 주문에 대해 승인 또는 거절을 처리한다.

#### 4.4.1 접수된 주문 목록
- `RESERVED` 상태의 주문만 목록으로 표시한다.

#### 4.4.2 주문 승인
접수된 특정 주문을 승인하면, **재고 상황에 따라 자동으로 2가지 방식 중 하나로 처리**된다.

| 재고 상황 | 처리 방식 |
|---|---|
| 재고 충분 | 주문을 즉시 `CONFIRMED` 상태로 전환 |
| 재고 부족 | 생산 라인에 자동으로 등록, 주문 상태를 `PRODUCING`으로 전환 |

**비즈니스 규칙**
- 승인 시점 기준의 재고 현황을 보고 필요한 부족분을 산정한다. (생산에 걸리는 시간은 고려하지 않고, 오직 승인 시점의 현재 재고만을 기준으로 부족 여부를 판단)
- 부족분 = 주문 수량 − 현재 재고
- 재고가 부족하여 생산 큐에 등록된 주문은 이후 취소하거나 중간에 변경할 수 없다. 큐에 들어온 이상 반드시 생산이 완료된다.
- 동일한 시료에 대한 주문이라도 각각 별도의 생산 작업으로 처리한다(작업을 병합하지 않음).

#### 4.4.3 주문 거절
- 접수된 특정 주문을 거절하면 즉시 `REJECTED` 상태로 전환된다.

### 4.5 모니터링
주문에 따른 시료 재고 정보를 담당자가 한눈에 파악할 수 있도록 제공하는 기능.

#### 4.5.1 주문량 확인
- 상태별(`RESERVED` / `CONFIRMED` / `PRODUCING` / `RELEASED`) 주문 건수를 확인한다.
- `REJECTED`는 유효한 주문이 아니므로 집계에서 제외한다.

#### 4.5.2 재고량 확인
- 각 시료별 현재 재고 수량을 확인한다.
- 주문 대비 재고 수량에 따라 아래와 같이 상태를 표기한다.

| 상태 | 의미 |
|---|---|
| 여유 | 주문 대비 재고 충분 |
| 부족 | 주문 대비 재고 수량 부족 |
| 고갈 | 재고 수량이 0인 상태 |

### 4.6 생산 라인
생산 라인은 시료 하나를 생산하는 설비 흐름을 의미하며, **시스템 내 생산 라인은 단일(1개) 라인**이다.
해당 라인은 시료를 한 번에 하나씩만 생산하며, 여러 주문의 생산 요청은 하나의 FIFO 큐에 쌓여 순차적으로 처리된다.
생산 라인은 **주문이 들어온 시료에 대해서만** 동작한다.

#### 4.6.1 생산량/생산시간 계산
- **실 생산량** = `ceil(부족분 / 수율)` — 수율에 따른 불량을 고려하여 목표 부족분을 채우기 위해 실제로 생산해야 하는 수량
- **총 생산 시간** = `평균 생산시간 × 실 생산량`
- **남는 재고(잉여분)** = 실 생산량 − 부족분 (수율로 인해 목표보다 더 생산된 여유분은 재고로 남는다)
- 생산이 완료되면 해당 주문 상태는 `PRODUCING → CONFIRMED`로 전환된다.
- 생산 완료 시점에 재고를 즉시 갱신(증가)한다.

**재고 확인/갱신 시점 (확정)**
| 시점 | 동작 |
|---|---|
| 주문 승인 시 | 물리적으로 재고를 차감하는 것이 아니라, 재고 부족/충분 여부를 정확히 판단하기 위해 최신 재고 값을 조회(재확인)한다 |
| 생산 완료 시 | 실 생산량만큼 재고를 실제로 증가시킨다 |
| 출고 처리 시 | 출고 수량만큼 재고를 실제로 차감한다 |

즉, 물리적인 재고 증감은 **생산 완료 시(+)** 와 **출고 시(-)** 두 시점에서만 발생하며, 주문 승인 시점은 판단을 위한 조회 용도로만 재고를 확인한다.

#### 4.6.2 생산 현황 표기
- 현재 생산 중인 시료에 대한 정보(주문 정보, 현재까지의 생산량 등)를 표시한다. 표기 정보의 구체적 수준은 자유롭게 결정한다.

#### 4.6.3 대기 주문 확인 (생산 큐)
- 생산 라인의 대기열(생산 큐)을 이용하여, 생산 작업을 대기 중인 목록을 출력한다.
- **스케줄링 전략: FIFO(선입선출)**

#### 4.6.4 실시간성 및 영속성 요구사항 (중요)
- 생산은 실제 경과 시간에 맞춰 진행되어야 한다. 예를 들어 예상 소요 시간이 10분이라면, 10분이 지난 뒤 조회 시 해당 작업은 완료 상태여야 한다.
- 애플리케이션이 실행 중이 아니어도(종료되었다가 재시작하더라도) 생산 경과 시간은 계속 흐른 것으로 간주해야 한다. 즉, 백그라운드로 계속 실행될 필요 없이, 재시작 시점의 현재 시각을 기준으로 그동안 완료됐어야 할 생산 큐 항목들을 모두 반영(정산)해야 한다.

### 4.7 출고 처리
재고가 충분해져 `CONFIRMED` 상태가 된 주문에 대해 출고를 처리한다.

- 출고 가능한 주문 목록은 `CONFIRMED` 상태의 주문에 한해서만 표시한다.
- 특정 주문에 대해 출고를 실행하면 주문 상태가 `CONFIRMED → RELEASED`로 전환된다.
- **부분 출고는 지원하지 않는다.** 출고는 항상 해당 주문의 전체 수량을 한 번에 처리하며, 별도의 출고 수량 입력 없이 주문 수량 전체가 그대로 출고된다.

## 5. 데이터 모델 요약

### 5.1 시료(Sample)
| 필드 | 타입/설명 |
|---|---|
| 시료 ID | 고유 식별자 |
| 이름 | 시료명 |
| 평균 생산시간 | 개당 평균 생산 시간(분 등) |
| 수율 | 0~1 사이의 비율 |
| 현재 재고 | 수량(정수) |

### 5.2 주문(Order)
| 필드 | 타입/설명 |
|---|---|
| 주문번호 | 고유 식별자 (예: ORD-YYYYMMDD-XXXX) |
| 시료 ID | 참조하는 시료 |
| 고객명 | 주문 고객 |
| 주문 수량 | 수량(정수) |
| 상태 | RESERVED / REJECTED / PRODUCING / CONFIRMED / RELEASED |
| 생성 일시 | 주문 접수 시각 |

### 5.3 생산 큐 항목(ProductionQueueItem)
| 필드 | 타입/설명 |
|---|---|
| 연결 주문번호 | 대응하는 주문 |
| 부족분 | 주문 수량 − 승인 시점 재고 |
| 실 생산량 | ceil(부족분 / 수율) |
| 총 생산 시간 | 평균 생산시간 × 실 생산량 |
| 생산 시작 시각 | 큐 진입/생산 시작 시각 |
| 예상 완료 시각 | 생산 시작 시각 + 총 생산 시간 |

## 6. 비기능 요구사항

### 6.1 저장소(Repository) 범위
- 미션에서 요구하는 4개의 PoC(MVC 스켈레톤 코드, 데이터 영속성 처리, 데이터 모니터링 Tool, Dummy 데이터 생성 Tool)는
  **이 저장소(SampleOrderSystem)에서 개발하지 않는다.** 각 PoC는 별도의 독립된 Repository에서 개발한다.
- 이 저장소는 4개 PoC에서 검증된 구조/방식을 참고하여 **반도체 시료 생산주문관리 시스템 본 프로젝트**만을 구현하는 데 집중한다.
- PoC 저장소가 먼저 만들어지면, 해당 GitHub 저장소 URL을 참고 자료로 확인하되 코드를 이 저장소로 그대로 옮겨오지 않고 본 프로젝트의 구조에 맞게 구현한다.

## 7. 용어 정리
| 용어 | 정의 |
|---|---|
| 시료(Sample) | 반도체 웨이퍼 공정으로 생산되는 제품 단위 |
| 수율 | 정상 생산된 시료 수 / 총 생산 시료 수 |
| 부족분 | 주문 수량 − 승인 시점의 현재 재고 |
| 실 생산량 | 수율을 고려해 부족분을 충족하기 위해 실제로 생산해야 하는 수량, `ceil(부족분 / 수율)` |
| FIFO | 생산 큐의 스케줄링 전략(선입선출) |
