# 커밋 규칙 (Commit Rules)

이 문서는 이 저장소에서 커밋을 생성할 때 반드시 지켜야 하는 규칙을 정의한다.
과제 요구사항상 커밋 이력이 평가 대상이므로, 아래 규칙을 어기고 커밋하지 않는다.

## 1. 커밋 단위 (Atomic Commit)

- 커밋은 **의미적으로 하나의 변경**만 담는다. 여러 기능/수정을 한 커밋에 몰아넣지 않는다.
  - 예: "Model 추가" + "Controller 추가" + "테스트 추가"를 한 커밋으로 합치지 않는다.
- 하나의 커밋만으로 빌드가 되고(컴파일 가능), 관련된 테스트가 통과하는 상태여야 한다.
- 리팩터링과 기능 추가는 반드시 별도 커밋으로 분리한다.
- 포맷팅/공백 정리 같은 스타일 변경은 기능 변경과 섞지 않는다.

## 2. 커밋 전 필수 확인

커밋하기 전에 아래를 순서대로 확인한다.

1. `git status`로 변경/추가된 파일을 확인하고, 의도한 범위의 파일만 스테이징한다(`git add -A` 남용 금지, 파일 단위로 지정).
2. 관련 단위 테스트(GoogleTest)를 실행하여 통과를 확인한다.
3. 빌드가 깨지지 않는지 확인한다.
4. `.env`, 인증 정보, IDE 개인 설정(`*.user`, `.vs/`) 등 커밋되면 안 되는 파일이 포함되지 않았는지 확인한다.

## 3. 커밋 메시지 규칙

형식:
```
<type>: <한 줄 요약 (무엇을 왜 했는지, 명령형)>

<선택: 본문 — 배경/이유가 코드만으로 드러나지 않을 때만 작성>
```

### type 종류
| type | 의미 |
|---|---|
| feat | 새로운 기능 추가 |
| fix | 버그 수정 |
| refactor | 동작 변화 없는 구조 개선 |
| test | 테스트 추가/수정 |
| docs | 문서(PRD.md, CLAUDE.md 등) 추가/수정 |
| chore | 빌드 설정, 의존성, 기타 잡무성 변경 |

### 작성 원칙
- 제목은 "무엇을 했는지"보다 **왜 했는지/어떤 문제를 해결했는지** 중심으로 간결하게 작성한다.
- 이미 코드/식별자로 알 수 있는 내용을 그대로 나열하지 않는다.
- 한글로 작성하되, 타입 키워드(`feat`, `fix` 등)와 코드 식별자는 원문(영문) 그대로 둔다.

## 4. 금지 사항

- **절대 금지**: `--no-verify`, `--no-gpg-sign` 등으로 훅/검증을 우회하는 커밋
- **절대 금지**: `git commit --amend`로 이미 존재하는 커밋을 수정 (사용자가 명시적으로 요청한 경우 제외)
- **절대 금지**: `git push --force` (사용자가 명시적으로 요청하고 위험성을 인지한 경우 제외)
- **절대 금지**: 사용자 확인 없이 임의로 커밋을 생성하는 것 — 커밋 실행 전 항상 변경 내역과 커밋 메시지를 먼저 보여주고 진행 여부를 확인받는다.
- 대량 커밋(수십 개 파일, 여러 기능 동시 반영)을 한 번에 만들지 않는다.

## 5. 커밋 이력 예시

```
feat: 시료 등록 기능(Model/Controller) 추가

docs: PRD 기반 CLAUDE.md 작성

test: Sample 재고 계산 로직(ceil(부족분/수율)) 단위 테스트 추가

fix: 생산 큐 재시작 시 경과 시간 미반영 문제 수정
```

이처럼 기능 단위, 계층 단위로 잘게 나누어 커밋 이력을 남긴다.
